# 剧本杀生成中的 RAG 使用策略

## 一、背景与目标

当前 AI 生成剧本存在一个核心问题：内容容易偏向小说化，缺少剧本杀应有的玩家视角、角色任务、信息差、线索闭环、DM 流程和可推理结构。

项目计划引入知识库能力，由 admin 端管理知识库，web 端在生成流程中通过服务端 API 合理引用知识库内容。该方向可行，但不能把 RAG 当成完整剧本生成引擎。更合适的定位是：

- RAG 提供剧本杀创作规则、结构模板、案例抽象、反例和质检标准。
- 分阶段生成流程负责原创设计和成稿。
- 结构化中间产物负责保证可玩性和逻辑闭环。
- 质检与重写流程负责降低小说化和线索断裂风险。

因此，NarrLight 应实现的是“结构化知识库 + 分阶段生成 + 检索增强质检 + 剧本杀专用中间表示”，而不是传统的“切片检索 + 拼接 prompt + 一次性生成全文”。

## 二、传统 RAG 在剧本生成中的主要风险

### 2.1 切片破坏上下文

原始剧本被切成普通 chunk 后，容易丢失上下文关系。例如某段角色独白离开了所属角色、幕次、线索作用和最终真相后，就只剩下孤立文本。模型拿到这种片段，无法判断它在完整剧本中的功能，生成结果容易变成碎片化模仿。

典型后果：

- 人物关系不连贯。
- 线索作用不清晰。
- 真相、误导、隐藏信息之间缺少闭环。
- 检索到的片段看似相关，但对当前生成阶段没有帮助。

### 2.2 RAG 本质上优化 prompt，不负责创新

RAG 的核心能力是把外部信息放进上下文，并不能天然产生原创诡计、原创人物关系或原创推理结构。如果依赖剧本文本切片来驱动创作，模型更容易复用已有桥段，而不是设计新的机制。

典型后果：

- 核心诡计与已有剧本高度相似。
- 人物关系、凶案结构、反转方式同质化。
- 生成结果像“参考资料改写”，不是原创剧本杀产品。

### 2.3 大量切片堆叠降低生成质量

检索结果越多，prompt 中的噪声越多。大量片段同时注入会抢占上下文预算，也会让模型无法判断优先级。

典型后果：

- 风格混杂。
- 任务约束被稀释。
- 模型被参考片段带偏。
- 角色本、线索卡、DM 手册之间口吻和结构不一致。

### 2.4 原始切片中存在大量低价值内容

电子版剧本里会有很多对生成无效的内容，例如长篇氛围描写、重复心理活动、低信息密度对白、格式残留、页眉页脚和无结构段落。这些内容直接入库，会污染召回质量。

典型后果：

- 召回结果“相似但无用”。
- 生成内容更小说化。
- 线索、任务、机制类信息被低价值叙事淹没。

### 2.5 风格污染

不同题材、时代、机制和阅读体验的剧本片段混在一起，容易导致生成风格不统一。例如本格推理、情感沉浸、欢乐机制、阵营本的写法差异很大，不能无差别召回。

### 2.6 抄袭和近似复刻风险

如果知识库直接保存大量原剧本文字，模型可能复用原文表达、关键桥段、角色关系、作案手法或复盘结构。即使不逐字复制，也可能产生高相似度内容。

### 2.7 阶段错配

生成角色本时召回复盘，生成线索卡时召回背景叙事，生成 DM 流程时召回玩家私密文本，都会降低质量。RAG 必须理解当前生成阶段，而不是只按语义相似度召回。

### 2.8 无法自动保证推理闭环

RAG 可以提供参考材料，但不能自动证明“玩家依据已获得线索可以推理出真相”。推理闭环必须通过结构化矩阵、规则校验和专门的质检步骤保证。

## 三、RAG 应该做什么

RAG 在 NarrLight 中应服务于“剧本杀创作方法”和“质量控制”，而不是直接提供成稿正文。

适合使用 RAG 的场景：

- 检索剧本杀结构规则。
- 检索分幕推进模板。
- 检索角色任务、隐藏动机和信息差设计模式。
- 检索线索卡设计原则。
- 检索误导线索、关键线索、闭环线索的设计规则。
- 检索 DM 流程、控场节点、搜证节奏和投票复盘规则。
- 检索反小说化规则。
- 检索质量评分标准。
- 检索经过脱敏、抽象后的优秀案例模式。
- 在生成后作为审稿资料，辅助发现结构缺口。

推荐知识形态：

```json
{
  "type": "角色信息释放模式",
  "stage": "第一幕",
  "genre": "本格推理",
  "pattern": "角色只知道自己与死者的旧怨，不知道该旧怨被第三人利用",
  "use_case": "制造合理嫌疑，同时避免过早暴露真凶",
  "constraints": [
    "不能同时给出第三人的完整行动线",
    "必须保留一个可被后续线索验证的矛盾点"
  ],
  "risk": "如果第一幕披露利用关系，会导致真凶过早暴露"
}
```

## 四、RAG 不应该做什么

不建议在 NarrLight 中使用 RAG 做以下事情：

- 直接检索原始剧本正文并生成整本剧本。
- 一次性召回大量 chunk 塞进 prompt。
- 让 web 端直接访问知识库或向量库。
- 把完整电子版剧本作为可被模型模仿的样例库。
- 只按向量相似度召回，不做类型、题材、人数、阶段过滤。
- 用 RAG 替代原创案件设计。
- 用 RAG 替代线索闭环校验。
- 在同一个生成步骤里混用角色本、复盘、线索卡、DM 手册等不同类型资料。
- 将低信息密度的氛围描写、心理描写、无效对白直接入库。

## 五、问题解决方式

### 5.1 用层级结构化知识替代普通切片

原始资料入库前先解析为层级结构，而不是直接按字数切片。

建议结构：

- 剧本级摘要。
- 世界观摘要。
- 角色列表。
- 角色关系图。
- 角色目标、秘密、谎言和嫌疑点。
- 分幕结构。
- 时间线。
- 线索列表。
- 每条线索的作用。
- 凶案机制。
- DM 流程。
- 投票与复盘结构。

检索时返回片段本身，还要返回父级上下文：

- 所属剧本摘要。
- 所属模块。
- 所属角色。
- 所属幕次。
- 前后摘要。
- 该片段在完整剧本中的功能。

### 5.2 先原创设计，再用 RAG 审稿和补强

生成流程不应先检索样例再照着写，而应先生成原创方案：

- 原创题材。
- 原创核心诡计。
- 原创人物关系。
- 原创时间线。
- 原创线索闭环。
- 原创玩家目标。

之后再使用 RAG 检查和补强：

- 机制是否成熟。
- 玩家是否有足够可盘问点。
- 线索是否过早泄露真相。
- 非凶手角色是否有参与感。
- DM 流程是否能实际带本。

### 5.3 分阶段少量召回

每个生成阶段只召回对应类型知识，建议每次 3-6 条高置信结果。

阶段召回策略：

| 生成阶段 | 可召回知识类型 |
|---|---|
| 立项与题材 | 类型规则、玩家体验、人数时长、难度参考 |
| 案件骨架 | 诡计机制、时间线结构、作案条件设计 |
| 角色设计 | 角色目标、隐藏动机、信息差、嫌疑点模板 |
| 线索设计 | 线索功能、误导方式、证据链闭环 |
| 分幕推进 | 信息释放节奏、冲突升级、搜证节点 |
| 玩家本写作 | 角色视角、私密信息、行动目标、可盘问点 |
| DM 手册 | 主持流程、控场提醒、投票与复盘节点 |
| 质检重写 | 反小说化规则、逻辑闭环评分、反例 |

### 5.4 入库前做清洗、筛选和抽象

知识库不应保存未经处理的全文切片。入库流程应包含：

- 去除页眉页脚、乱码、重复段落和格式残留。
- 删除低信息密度氛围描写。
- 删除与结构无关的心理独白。
- 提取角色目标、秘密、动机、谎言和信息差。
- 提取线索作用，而不只是线索文本。
- 提取 DM 流程节点。
- 提取“好用模式”和“风险反例”。
- 对原文做摘要化、脱敏化和抽象化，降低复刻风险。

### 5.5 建立剧本杀专用中间表示

在生成正文前，必须先生成结构化中间产物：

- 角色信息矩阵。
- 角色目标矩阵。
- 线索矩阵。
- 时间线矩阵。
- 谎言与真相矩阵。
- 信息释放矩阵。
- 玩家可推理链路。
- DM 流程表。

只有这些结构通过检查后，才进入玩家本、线索卡、DM 手册和复盘正文生成。

### 5.6 增加反小说化质检器

每个模块生成后都应进行小说化风险评分。

评分维度：

- 大段连续叙事比例。
- 心理描写和环境描写比例。
- 无功能段落比例。
- 是否缺少玩家任务。
- 是否缺少可盘问信息。
- 是否缺少线索引用。
- 是否缺少阶段节点。
- 是否提前讲透真相。
- 是否缺少 DM 可执行流程。

建议阈值：

| 模块 | 小说化风险阈值 |
|---|---:|
| 角色本 | <= 40 |
| 线索卡 | <= 15 |
| DM 手册 | <= 10 |
| 复盘 | <= 30 |
| 开场背景 | <= 50 |

阈值不通过时，进入自动重写，而不是直接交付给用户。

## 六、在 NarrLight 现有系统中的实现方式

### 6.1 系统边界

推荐边界：

- `apps/admin`：知识库管理、资料入库、标签维护、启停和权重管理。
- `apps/web`：用户发起生成、查看生成结果、编辑剧本内容。
- 服务端 API：负责检索、权限、安全、prompt 组装和生成编排。
- Supabase：存储知识库文档、结构化知识、向量、生成任务和引用记录。

web 端不应直接调用向量库。web 端应调用生成 API，由服务端根据当前生成阶段决定是否检索、检索什么、引用多少。

### 6.2 Admin 端知识库能力

admin 端建议增加“创作知识库”模块，能力包括：

- 创建知识库。
- 上传电子版资料。
- 查看解析结果。
- 管理资料类型。
- 管理题材、人数、难度、阶段、模块等标签。
- 查看抽象后的知识条目。
- 人工编辑摘要、规则、风险和适用场景。
- 设置启用状态和召回权重。
- 查看知识被哪些生成任务引用。
- 标记低质量、重复或高风险知识。

知识类型建议分为：

- 结构规则。
- 角色设计模式。
- 线索设计模式。
- 时间线设计模式。
- DM 流程模板。
- 复盘结构模板。
- 反小说化规则。
- 质量评分标准。
- 案例抽象。
- 反例。

### 6.3 Web 端生成调用方式

web 端不需要感知底层知识库细节，只传递生成目标和当前阶段上下文。

示例流程：

1. 用户在 web 端提交题材、人数、时长、难度和偏好。
2. 服务端创建生成任务。
3. 服务端生成原创案件骨架。
4. 服务端按阶段检索相关规则和模式。
5. 服务端生成结构化中间产物。
6. 服务端执行逻辑闭环和反小说化质检。
7. 不合格模块自动重写。
8. 合格后生成玩家本、线索卡、DM 手册和复盘。
9. 记录本次使用的知识条目和质检结果。

### 6.4 API 设计建议

建议 API 不暴露通用检索给 web 端，而是暴露面向生成阶段的接口。

示例：

```ts
POST /api/generation/tasks
POST /api/generation/tasks/:taskId/stages/:stage/run
POST /api/generation/tasks/:taskId/quality-check
GET  /api/generation/tasks/:taskId/knowledge-usages
```

内部服务可拆分为：

```ts
retrieveKnowledge({
  stage,
  genre,
  playerCount,
  difficulty,
  categories,
  limit
})

buildGenerationPrompt({
  stage,
  userBrief,
  scriptState,
  knowledgeItems,
  qualityRules
})

runQualityCheck({
  moduleType,
  generatedContent,
  scriptState,
  qualityRules
})
```

### 6.5 数据表建议

可以在现有 Supabase 体系中新增以下表。字段命名后续应以实际迁移为准。

```text
knowledge_bases
knowledge_documents
knowledge_items
knowledge_embeddings
knowledge_tags
generation_knowledge_usages
generation_quality_reports
```

核心字段建议：

```text
knowledge_items
- id
- knowledge_base_id
- document_id
- title
- content
- summary
- category
- module_type
- stage
- genre
- player_count_min
- player_count_max
- difficulty
- source_context
- abstraction_level
- enabled
- weight
- metadata
- created_at
- updated_at
```

`abstraction_level` 用于区分：

- `raw_excerpt`：原文摘录，默认不直接用于生成。
- `summary`：摘要。
- `pattern`：抽象模式。
- `rule`：规则。
- `anti_pattern`：反例。
- `quality_metric`：评分项。

生成时应优先召回 `pattern`、`rule`、`anti_pattern` 和 `quality_metric`，谨慎使用 `raw_excerpt`。

### 6.6 Prompt 组装原则

prompt 优先级建议固定为：

1. 系统级生成规范。
2. 当前阶段任务。
3. 当前剧本结构化状态。
4. 必须遵守的质量规则。
5. 少量检索知识。
6. 输出格式要求。

知识片段必须附带用途说明，而不是裸文本堆叠。

示例：

```text
以下知识仅用于设计参考，不得复刻人物、桥段、台词或作案手法。

知识 1
类型: 线索设计模式
适用阶段: 第二幕
用途: 设计误导线索，但保留后续反证空间
内容: ...
```

## 七、推荐迭代路线

### 7.1 第一阶段：规则型知识库

目标是快速降低小说化风险。

- 先人工整理 20-50 条剧本杀结构规则。
- 建立反小说化评分标准。
- 在生成链路中加入阶段化 prompt 和质检重写。
- admin 端先支持知识条目的增删改查、启用和权重。

#### 7.1.1 第一阶段范围

第一阶段不处理大规模电子版剧本上传和自动解析，也不做完整向量检索系统。优先用人工录入的高质量规则和模板，快速验证“规则型知识库 + 分阶段生成 + 反小说化质检”是否能改善生成质量。

本阶段只做：

- 规则型知识条目管理。
- 生成阶段与知识类型绑定。
- 服务端按阶段读取少量启用知识。
- prompt 中注入结构规则和反小说化规则。
- 对生成结果做小说化评分。
- 评分不通过时自动重写一次。
- 记录本次生成使用过的知识条目。

本阶段不做：

- 不做电子版剧本批量上传。
- 不做原文 chunk 入库。
- 不做 embedding 和向量召回。
- 不做复杂知识图谱。
- 不做多轮自动优化权重。
- 不做模型微调。

#### 7.1.2 可验证成功标准

第一阶段完成后，应能验证以下结果：

- admin 端可以维护 20-50 条规则型知识。
- 每条知识可设置类型、适用阶段、题材、启用状态和权重。
- web 端发起生成时，服务端能按当前阶段引用对应知识。
- 生成结果中角色本、线索卡、DM 手册和复盘的输出格式明显区分。
- 小说化质检能识别大段连续叙事、无功能描写和缺少玩家任务的问题。
- 质检不通过时，系统能触发一次重写。
- 后台能追踪某次生成使用了哪些知识。

#### 7.1.4 Admin 端页面

admin 端新增“创作知识库”模块，第一阶段只需要三个页面。

知识条目列表：

- 表格展示标题、类型、阶段、题材、权重、启用状态、更新时间。
- 支持按类型、阶段、题材、启用状态筛选。
- 支持启用、停用、编辑、删除。
- 支持复制条目，方便快速建立相似规则。

知识条目编辑：

- 标题。
- 内容。
- 类型。
- 适用阶段。
- 适用题材。
- 适用人数范围。
- 难度。
- 权重。
- 启用状态。
- 备注 metadata。

生成引用记录：

- 按生成任务查看引用过的知识。
- 展示引用阶段、模块、知识标题、引用原因。
- 展示小说化质检分数和是否触发重写。

#### 7.1.5 Web 端生成流程改造

web 端仍然只负责提交用户生成需求，不直接访问知识库。

推荐流程：

```text
用户提交生成需求
-> 创建 generation_task
-> 生成原创案件骨架
-> 服务端按 stage 检索规则型知识
-> 组装阶段 prompt
-> 生成对应模块
-> 运行小说化质检
-> 不通过则带问题重写一次
-> 保存生成结果和质检报告
-> 返回给 web 端展示
```

第一阶段优先接入这些模块：

- `case_core`：案件骨架。
- `characters`：角色目标和信息差。
- `clues`：线索矩阵。
- `player_script`：玩家本。
- `dm_manual`：DM 手册。
- `review`：质检与重写。

#### 7.1.6 服务端检索逻辑

第一阶段先不用向量检索，使用确定性规则过滤，保证可控。

检索条件：

- `enabled = true`
- `stage = 当前生成阶段` 或 `stage = review`
- `category` 属于当前阶段允许类型
- `genre` 为空或匹配当前题材
- `player_count_min/max` 为空或覆盖当前人数
- 按 `weight desc, updated_at desc` 排序
- 每阶段最多取 3-6 条

示例伪代码：

```ts
async function retrieveStageKnowledge(input: {
  stage: GenerationStage
  genre?: string
  playerCount?: number
  limit?: number
}) {
  return queryKnowledgeItems({
    enabled: true,
    stage: [input.stage, "review"],
    genre: [input.genre, null],
    playerCount: input.playerCount,
    orderBy: ["weight desc", "updated_at desc"],
    limit: input.limit ?? 6
  })
}
```

#### 7.1.7 Prompt 组装方式

第一阶段应明确告诉模型：知识只用于规则约束，不用于复刻内容。

推荐结构：

```text
你正在生成剧本杀的【当前阶段】模块。

用户需求:
...

当前剧本结构状态:
...

必须遵守的剧本杀规则:
1. ...
2. ...

反小说化规则:
1. 不要写大段连续心理描写。
2. 每段内容必须服务于角色信息、玩家目标、线索或 DM 流程。
3. 不得提前讲透真相。

参考知识:
- 类型: ...
  用途: ...
  内容: ...

输出格式:
...
```

#### 7.1.8 小说化质检规则

第一阶段可以先用 LLM 质检，不必做复杂 NLP 规则。

质检输出建议固定为 JSON：

```json
{
  "score": 28,
  "risk_level": "low",
  "rewrite_required": false,
  "issues": [
    {
      "type": "missing_player_goal",
      "severity": "medium",
      "message": "第二个角色缺少明确的本轮目标"
    }
  ],
  "rewrite_instructions": [
    "为第二个角色补充可执行目标和可盘问点"
  ]
}
```

评分建议：

- `0-30`：低风险，可通过。
- `31-60`：中风险，建议重写。
- `61-100`：高风险，必须重写。

第一阶段可以只自动重写一次，避免生成任务无限循环。重写后仍不通过时，保存质检报告，并在 web 端提示用户该模块存在质量风险。

#### 7.1.9 初始规则样例

第一阶段建议先人工录入以下类型规则：

- 角色本每一幕必须包含“已知事实、隐瞒信息、当前目标、可盘问对象”。
- 线索卡必须标记线索功能：关键线索、误导线索、补强线索、反证线索。
- DM 手册必须按时间顺序描述主持动作，禁止写成小说旁白。
- 复盘必须包含真相、时间线、证据链和每个角色的行为解释。
- 开场背景允许文学化，但不得超过整本内容的固定比例。
- 非凶手角色也必须拥有独立秘密、动机和阶段性目标。
- 真凶角色不得在早期文本中暴露完整作案逻辑。
- 每个关键结论至少需要两条线索支撑。
- 每个角色至少拥有一个会被其他玩家质疑的矛盾点。
- 玩家本中的叙事段落必须服务于信息释放，不能只做氛围描写。

#### 7.1.10 开发顺序

建议按以下顺序实现：

1. 新增 Supabase migration，建立三张最小表。
2. 在 shared 层定义知识类型、阶段、质检报告类型。
3. 在 admin 端实现知识条目列表和编辑页面。
4. 在服务端实现 `retrieveStageKnowledge`。
5. 在生成链路中接入阶段化知识注入。
6. 实现小说化质检 prompt 和 JSON 解析。
7. 实现不通过时自动重写一次。
8. 保存 `generation_knowledge_usages` 和 `generation_quality_reports`。
9. 在 admin 端补充引用记录页面。
10. 用 3-5 个生成案例人工对比改造前后质量。

#### 7.1.11 第一阶段验收用例

验收时至少覆盖：

- 生成 6 人本格推理本，检查是否产生角色目标、线索矩阵、DM 流程。
- 生成情感沉浸本，检查是否保留适度叙事但不丢失任务和线索。
- 禁用某条知识后，再次生成不应引用该知识。
- 提高某条知识权重后，同阶段召回排序应提升。
- 故意输入小说化内容，质检应给出中高风险并触发重写。
- 生成完成后，admin 能看到知识引用记录和质检报告。

### 7.2 第二阶段：资料解析与结构化抽取

目标是利用现有电子版剧本数据，但避免直接复刻。

- 支持上传资料。
- 自动解析文本。
- 自动抽取角色、线索、幕次、时间线和 DM 流程。
- 生成摘要、模式、规则和反例。
- admin 人工审核后入库。

### 7.3 第三阶段：分阶段 RAG 生成

目标是让知识库在不同生成阶段发挥不同作用。

- 案件骨架阶段召回机制类知识。
- 角色阶段召回角色目标和信息差模式。
- 线索阶段召回线索闭环和误导模式。
- DM 阶段召回流程规则。
- 质检阶段召回评分标准和反例。

### 7.4 第四阶段：质量闭环

目标是让知识库持续优化。

- 记录每次生成引用了哪些知识。
- 记录质检分数。
- 记录用户重写、废弃、采纳行为。
- 在 admin 端查看知识效果。
- 降低低价值知识权重，提升高价值知识权重。

## 八、结论

RAG 仍然可以用于剧本杀生成，但不能采用传统的全文切片检索方式。对 NarrLight 来说，RAG 的合理定位是“创作知识增强”和“生成质量控制”。

最终方案应遵循以下原则：

- 不用 RAG 直接生成完整剧本。
- 不把大量原文切片塞进 prompt。
- 不让 web 端直接访问知识库。
- 用 admin 管理结构化创作知识。
- 用服务端按生成阶段检索少量高质量知识。
- 先生成原创结构，再用 RAG 审稿和补强。
- 通过结构化矩阵和质检器降低小说化风险。

这样既能利用现有电子版剧本数据，又能避免上下文断裂、缺少创新、堆料降质、无效数据污染和版权复刻风险。
