# NarrLight 项目宪法

## 核心原则

### I. Skill 优先执行

在编写或修改代码之前，代理 MUST 检查可用 Skill 列表，并调用与当前任务意图匹配的任何 Skill。例如：页面设计使用 UI/UX Skill、接口设计使用 API 设计 Skill、数据库设计使用数据库设计 Skill。 rationale：复用已建立的 Skill 可以保持 consistency、减少重复，并确保领域最佳实践被自动应用。

### II. 视觉与设计完整性

所有用户界面 MUST 遵守以下视觉规则：

- 严禁使用蓝紫渐变色（例如 `from-blue-500 to-purple-600`）。
- 严禁使用字体图标；应使用 SVG 或基于图片的图标。
- 所有设计 MUST 在组件和页面之间保持一致的样式、间距和布局语言。

rationale：统一的视觉语言可以减少"生成感"、提升可访问性，并让用户感受到产品的整体性。

### III. 技术栈锁定

除非用户明确覆盖，项目 MUST 使用批准的技术栈：

- 前端：Next.js、TypeScript、Ant Design、Tailwind CSS 4。
- 构建工具：Vite 8。
- 样式方案：Tailwind CSS 是默认样式方案。CSS-in-JS 库（例如 Styled Components）和 CSS Modules MUST NOT 使用，除非明确要求。

rationale：锁定的技术栈可以减少决策负担、确保互操作性，并使构建和维护成本可预测。

### IV. 代码质量与简洁性

所有代码 MUST 遵循以下质量规则：

- 选择能够解决问题的最简单实现。严禁过度设计。
- TypeScript 中严禁使用 `any` 类型。类型不明时使用 `unknown`，确实无值时使用 `never`。
- 变量名和函数名 MUST 具有自解释性。严禁使用缩写（例如使用 `getUserData`，而非 `getData` 或 `gud`）。

rationale：简单、类型安全、命名清晰的代码更易于 review、测试和长期维护。

### V. 行为、伦理与可访问性

所有生产代码 MUST 对用户和数据负责：

- 严禁使用 "Lorem Ipsum" 或通用占位文本。应使用符合业务语境的真实占位内容，或直接向用户询问具体内容。
- 生产代码中严禁硬编码 Mock 数据。MUST 实现真实的 Loading、Error 和 Empty 状态，除非明确要求除外。
- 严禁在前端代码中硬编码 API Key 或敏感凭证。
- 所有交互元素 MUST 包含正确的 ARIA 属性，并 MUST 支持通过 Tab 键进行键盘导航。

rationale：真实的内容、安全的凭证处理以及完整的键盘可访问性可以保护用户，并体现专业工程标准。

### VI. 交互与响应标准

所有用户交互 MUST 满足以下响应和安全规则：

- 点击、悬停和焦点变化的视觉反馈 MUST 在 100 毫秒内完成。
- 执行破坏性操作（例如删除数据或覆盖文件）之前，MUST 先向用户列出明确计划并取得明确确认。

rationale：快速反馈带来流畅的用户体验，确认机制可以防止误操作导致的数据丢失。

### VII. 文档语言规范

项目生成的所有自然语言文档 MUST 使用中文。这包括但不限于规格说明（spec.md）、实现计划（plan.md）、任务列表（tasks.md）、README、设计文档、用户故事、验收标准和注释。代码中的变量名、函数名、类型名和文件路径等技术标识符可保留英文，但面向人类阅读的说明性文本 MUST 使用中文。

rationale：统一使用中文可以降低团队成员的阅读成本，避免中英混杂导致的歧义，并确保项目文档对中文母语用户友好。

## 技术约束

原则 III 中定义的技术栈是强制性的。仅在某个功能确实需要时，MAY 向列表中添加新技术，并且 MUST 记录理由。未经用户批准，代理 MUST NOT 引入替代构建工具、样式方案或 UI 库。

## 开发流程与质量门

每次实现 MUST 通过以下质量门：

- 在编码开始前，验证是否存在匹配的 Skill 并已调用。
- 确认所选方案遵守上述视觉、技术和质量原则。
- 对于破坏性变更，MUST 先向用户呈现计划并取得确认。
- 在标记任务完成前，审查代码中是否存在 `any` 用法、禁用渐变、字体图标、缺失 ARIA 属性以及硬编码密钥。

## 治理

本项目宪法取代 NarrLight 项目中的所有其他编码约定。宪法修正案 MUST 记录在 `.specify/memory/constitution.md` 中，MUST 按照语义化版本控制进行版本升级，并且 MUST 包含针对进行中工作的迁移或合规计划。所有 Pull Request 和代理会话 SHOULD 在结束前验证是否符合这些原则。
